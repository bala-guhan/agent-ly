OVERVIEW
================================================================================

The 2care API provides secure, standards-based access to healthcare data and functionality across the 2care platform. Our RESTful API enables developers to integrate 2care capabilities into their applications, build custom workflows, and create innovative healthcare solutions.

Key Features:
- RESTful architecture following industry best practices
- OAuth 2.0 and OpenID Connect authentication
- FHIR R4 compliance for clinical data
- Comprehensive webhook support for real-time notifications
- Rate limiting and throttling for system protection
- Detailed error handling and logging
- Sandbox environment for development and testing
- Comprehensive SDKs for popular languages

Base URL:
- Production: https://api.2care.com/v3
- Sandbox: https://sandbox-api.2care.com/v3

API Documentation Portal:
- Interactive API documentation: https://developers.2care.com
- Developer community: https://community.2care.com
- Status page: https://status.2care.com

AUTHENTICATION AND AUTHORIZATION
================================================================================

OAuth 2.0 Implementation:

The 2care API uses OAuth 2.0 for authentication and authorization. We support multiple OAuth 2.0 grant types depending on your use case:

Authorization Code Grant (Recommended for web applications):

Step 1: Direct user to authorization endpoint
GET https://auth.2care.com/oauth/authorize
  ?response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=YOUR_REDIRECT_URI
  &scope=patient:read patient:write
  &state=RANDOM_STATE_STRING

Step 2: Exchange authorization code for access token
POST https://auth.2care.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTHORIZATION_CODE
&redirect_uri=YOUR_REDIRECT_URI
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET

Response:
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "tGzv3JOkF0XG5Qx2TlKWIA",
  "scope": "patient:read patient:write"
}

Client Credentials Grant (For server-to-server applications):

POST https://auth.2care.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
&scope=system:read system:write

Refresh Token:

POST https://auth.2care.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=REFRESH_TOKEN
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET

Using Access Tokens:

All API requests must include the access token in the Authorization header:

GET https://api.2care.com/v3/patients/12345
Authorization: Bearer YOUR_ACCESS_TOKEN
Accept: application/json

Available Scopes:

Patient Data:
- patient:read - Read patient demographics and records
- patient:write - Create and update patient information
- patient:delete - Delete patient records (restricted)

Clinical Data:
- clinical:read - Read clinical observations, conditions, procedures
- clinical:write - Create and update clinical data
- medication:read - Read medication information
- medication:write - Prescribe and update medications

Appointments:
- appointment:read - View appointment information
- appointment:write - Schedule and modify appointments

Documents:
- document:read - Access clinical documents
- document:write - Upload and update documents

Analytics:
- analytics:read - Access reporting and analytics data

Administration:
- user:read - Read user information
- user:write - Manage user accounts
- system:admin - Full administrative access (highly restricted)

FHIR Search:

Search for patients by name:
GET https://api.2care.com/v3/fhir/Patient?name=Smith

Search for observations by patient and date:
GET https://api.2care.com/v3/fhir/Observation
  ?patient=example-patient-id
  &date=ge2025-01-01
  &_sort=-date

Search parameters support:
- Exact matches: ?name=Smith
- Partial matches: ?name:contains=Smi
- Date ranges: ?date=ge2025-01-01&date=le2025-12-31
- Multiple values: ?status=active,suspended
- Sorting: ?_sort=name, ?_sort=-date (descending)
- Pagination: ?_count=50&_offset=100
- Includes: ?_include=Patient:organization

WEBHOOKS
================================================================================

2care supports webhooks for real-time event notifications.

Setting Up Webhooks:

Create Webhook Subscription:
POST /v3/webhooks
Content-Type: application/json

{
  "url": "https://your-app.com/webhooks/2care",
  "events": [
    "patient.created",
    "patient.updated",
    "appointment.scheduled",
    "appointment.cancelled",
    "document.uploaded"
  ],
  "secret": "your_webhook_secret_for_verification"
}

Response: 201 Created
{
  "id": "webhook_abc123",
  "url": "https://your-app.com/webhooks/2care",
  "events": ["patient.created", "patient.updated", ...],
  "status": "active",
  "createdAt": "2026-01-20T15:30:00Z"
}

Webhook Payload Format:

POST https://your-app.com/webhooks/2care
Content-Type: application/json
X-2Care-Signature: sha256=abc123...
X-2Care-Event: patient.created
X-2Care-Delivery-Id: evt_xyz789

{
  "id": "evt_xyz789",
  "type": "patient.created",
  "createdAt": "2026-01-20T15:35:00Z",
  "data": {
    "object": {
      "id": "pat_1234567890",
      "mrn": "MRN-2026-001234",
      "firstName": "John",
      "lastName": "Smith",
      ... 
    }
  }
}

Verifying Webhook Signatures:

const crypto = require('crypto');

function verifyWebhookSignature(payload, signature, secret) {
  const hmac = crypto.createHmac('sha256', secret);
  const digest = 'sha256=' + hmac.update(payload).digest('hex');
  return crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(digest));
}

================================================================================
RATE LIMITING
================================================================================

Rate Limits:

Standard Tier:
- 1,000 requests per hour per API key
- Burst limit: 100 requests per minute

Professional Tier:
- 5,000 requests per hour per API key
- Burst limit: 500 requests per minute

Enterprise Tier:
- Custom limits based on agreement
- Dedicated rate limit pools

Rate Limit Headers:

Every API response includes rate limit information:

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1674234000

When rate limit is exceeded:

HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 3600

{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "API rate limit exceeded. Please retry after 3600 seconds.",
    "retryAfter": 3600
  }
}

Best Practices:
- Implement exponential backoff for retries
- Monitor rate limit headers
- Cache responses when appropriate
- Use webhooks instead of polling for real-time updates
- Contact support for higher limits if needed

================================================================================
ERROR HANDLING
================================================================================

Standard Error Response Format:

{
  "error": {
    "code": "invalid_request",
    "message": "The request is missing required parameter 'patientId'",
    "field": "patientId",
    "requestId": "req_abc123xyz789"
  }
}

HTTP Status Codes:

Success:
- 200 OK - Request successful
- 201 Created - Resource created successfully
- 204 No Content - Successful deletion

Client Errors:
- 400 Bad Request - Invalid request parameters
- 401 Unauthorized - Missing or invalid authentication
- 403 Forbidden - Insufficient permissions
- 404 Not Found - Resource not found
- 409 Conflict - Resource conflict (e.g., duplicate)
- 422 Unprocessable Entity - Validation error
- 429 Too Many Requests - Rate limit exceeded

Server Errors:
- 500 Internal Server Error - Server error
- 502 Bad Gateway - Upstream service error
- 503 Service Unavailable - Service temporarily unavailable
- 504 Gateway Timeout - Request timeout

Common Error Codes:

Authentication/Authorization:
- invalid_token - Access token is invalid or expired
- insufficient_scope - Token lacks required scope
- invalid_credentials - Invalid client credentials

Validation:
- invalid_request - Malformed request
- missing_parameter - Required parameter missing
- invalid_parameter - Parameter value is invalid
- validation_error - Data validation failed

Resource:
- resource_not_found - Requested resource doesn't exist
- resource_conflict - Resource already exists
- resource_deleted - Resource has been deleted

System:
- rate_limit_exceeded - Too many requests
- server_error - Internal server error
- service_unavailable - Service temporarily down

BEST PRACTICES
================================================================================

Security:
- Never expose API keys in client-side code
- Rotate API keys regularly
- Use environment variables for credentials
- Implement proper token storage on clients
- Validate webhook signatures
- Use HTTPS for all requests

Performance:
- Implement caching where appropriate
- Use pagination for large result sets
- Request only needed fields using field selection
- Use batch operations when updating multiple resources
- Implement connection pooling

Reliability:
- Implement retry logic with exponential backoff
- Handle rate limiting gracefully
- Monitor API status page
- Implement circuit breakers
- Log all API interactions for debugging

Data Integrity:
- Validate data before sending to API
- Handle partial failures in batch operations
- Implement idempotency for critical operations
- Use transaction IDs for request tracking
